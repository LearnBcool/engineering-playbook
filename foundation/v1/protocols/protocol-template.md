# Protocol Template

> Foundation v1

Every engineering protocol follows the same structure.

The purpose of this template is to standardize engineering conversations, ensuring that discussions remain focused, productive, and traceable.

A protocol defines **how engineering decisions are made**, not **what decisions should be made**.

---

# Protocol Information

**Protocol ID**

Unique identifier.

Example:

```
PR-000
```

---

**Protocol Name**

Human-readable protocol name.

Example:

```
Project Zero
```

---

**Related Stage**

The engineering stage where this protocol is executed.

Example:

```
Stage 00 — Project Zero
```

---

# Objective

Why does this protocol exist?

A protocol must have exactly one primary objective.

The objective should describe the engineering outcome expected from the discussion.

---

# Trigger

What event starts this protocol?

Examples:

- New project created.
- Architecture review requested.
- Sprint completed.
- Deployment approved.

Every protocol must have a clearly defined trigger.

---

# Participants

List all required participants.

For each participant define:

- Role
- Responsibility
- Required contribution

Only necessary participants should be included.

---

# Facilitator

Who leads the discussion?

The facilitator is responsible for:

- Keeping the meeting focused.
- Managing speaking order.
- Summarizing decisions.
- Closing the protocol.

The facilitator does not need to be the decision maker.

---

# Decision Authority

Who makes the final decision?

Engineering discussions should always have explicit decision ownership.

---

# Meeting Rules

Define the behavioral rules for the discussion.

Examples:

- Respect every participant.
- Challenge ideas, not people.
- Support claims with evidence.
- Avoid unrelated topics.
- Record important decisions.

Meeting rules apply throughout the protocol.

---

# Discussion Flow

Describe the sequence of discussion.

Each step should include:

- Step Name
- Objective
- Responsible Participant
- Expected Output

The discussion should progress through clearly defined states.

---

# Decision Process

Describe how decisions are reached.

Examples:

- Consensus
- Engineering Leader decision
- Evidence-based evaluation
- Majority vote

The decision process must be explicit.

---

# Conflict Resolution

Describe how disagreements are handled.

Possible approaches include:

- Request additional research.
- Escalate to the Engineering Leader.
- Record competing alternatives.
- Postpone the decision until more evidence exists.

Engineering disagreement is expected.

The protocol exists to resolve it constructively.

---

# Outputs

List every artifact produced by the protocol.

Examples:

- Engineering Decision Record (EDR)
- Project Charter
- Risk Register
- Architecture Proposal

Outputs become inputs for future stages.

---

# Completion Criteria

A protocol is complete only when:

- Every discussion step has been completed.
- Required outputs exist.
- Decisions are documented.
- Outstanding issues are recorded.
- The facilitator formally closes the protocol.

---

# Engineering Notes

Optional observations that provide context without modifying the protocol.

---

# Engineering Principle

A protocol succeeds when it produces clarity, alignment, and documented decisions.

Its success is measured by the quality of the engineering conversation, not by the speed at which it finishes.
