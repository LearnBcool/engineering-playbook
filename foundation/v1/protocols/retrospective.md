# Engineering Conversation Protocol (ECP-009)

> Retrospective Protocol
> Foundation v1

---

# Protocol Information

**Protocol ID**

ECP-009

---

**Protocol Name**

Retrospective

---

**Related Stage**

Stage 08 — Retrospective

---

# Objective

Reflect on the completed engineering cycle to identify improvements in processes, decisions, collaboration, and execution.

The purpose of this protocol is not to assign blame.

Its purpose is to improve the engineering system itself.

Continuous improvement is an engineering responsibility.

---

# Trigger

Start automatically after:

- Deployment has been completed.
- The project milestone has been officially closed.

---

# Participants

## Engineering Leader

Leads the retrospective.

Approves improvement actions.

---

## Problem Analyst

Evaluates whether the original problem was correctly understood.

---

## Research Engineer

Evaluates whether research sufficiently reduced uncertainty.

---

## Software Architect

Reviews architectural effectiveness.

---

## Backend Engineer

Reflects on backend implementation.

---

## Frontend Engineer

Reflects on frontend implementation.

---

## QA Engineer

Evaluates software quality and validation effectiveness.

---

## Infrastructure Engineer

Reviews operational readiness and deployment.

---

## Leadership Observer

Evaluates leadership performance throughout the project.

Provides structured leadership feedback.

---

# Facilitator

Engineering Leader

---

# Decision Authority

Engineering Leader

---

# Meeting Rules

- Focus on learning.
- Critique systems, not people.
- Support observations with evidence.
- Celebrate successes.
- Record improvement opportunities.
- Every improvement should be actionable.

---

# Discussion Flow

## Phase 1 — Project Review

Review the complete engineering lifecycle.

Expected Output

Project Summary

---

## Phase 2 — Success Analysis

Discuss:

- What worked well?
- What exceeded expectations?
- Which practices should become standards?

Expected Output

Success Register

---

## Phase 3 — Challenge Analysis

Discuss:

- What created unnecessary difficulty?
- Which assumptions proved incorrect?
- Which risks materialized?

Expected Output

Challenge Register

---

## Phase 4 — Root Cause Review

Analyze recurring problems.

Identify:

- Process issues
- Communication issues
- Technical issues
- Leadership issues

Expected Output

Root Cause Analysis

---

## Phase 5 — Methodology Review

Evaluate the engineering methodology itself.

Questions:

- Which stages created the most value?
- Which protocols need improvement?
- Which artifacts were unnecessary?
- Which artifacts were missing?
- Which decisions should be standardized?

Expected Output

Methodology Improvement Report

---

## Phase 6 — Leadership Review

Leadership Observer presents an evaluation covering:

- Decision quality
- Communication
- Team coordination
- Conflict management
- Learning culture

Expected Output

Leadership Assessment

---

## Phase 7 — Improvement Planning

Define concrete improvement actions.

For each action specify:

- Description
- Owner
- Priority
- Target version of the Foundation

Expected Output

Continuous Improvement Backlog

---

## Phase 8 — Foundation Evolution

Review whether improvements should update:

- Philosophy
- Constitution
- Methodology
- Engine
- Stages
- Protocols
- Roles
- Artifacts

Expected Output

Foundation Change Proposal

---

## Phase 9 — Closing Reflection

Engineering Leader summarizes the project's most important lessons.

Expected Output

Engineering Lessons Learned

---

# Conflict Resolution

If disagreement exists:

1. Review objective evidence.
2. Separate facts from opinions.
3. Document differing perspectives.
4. Prioritize learning over consensus.
5. Engineering Leader approves final conclusions.

---

# Outputs

This protocol produces:

- Project Summary
- Success Register
- Challenge Register
- Root Cause Analysis
- Methodology Improvement Report
- Leadership Assessment
- Continuous Improvement Backlog
- Foundation Change Proposal
- Engineering Lessons Learned

---

# Completion Criteria

The retrospective is complete only when:

- The engineering lifecycle has been reviewed.
- Improvement opportunities are documented.
- Leadership feedback has been delivered.
- Methodology improvements have been identified.
- Foundation evolution proposals have been recorded.

---

# Leadership Reflection

The Leadership Observer evaluates:

- Did leadership create psychological safety?
- Were difficult conversations handled constructively?
- Did the team focus on improvement rather than blame?
- Were lessons transformed into actions?
- Did the project improve the engineering system?

---

# Engineering Principle

Every completed project is an experiment.

Its greatest value is not only the software delivered, but the engineering knowledge gained.

Engineering excellence is achieved through continuous learning, disciplined reflection, and deliberate evolution.
