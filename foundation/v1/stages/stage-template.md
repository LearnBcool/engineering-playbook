# Stage Template

> Foundation v1

Every engineering stage follows the same structure.

The purpose of this template is to ensure consistency, predictability, and repeatability across every project executed using the Engineering Playbook.

No stage may introduce additional sections without explicit justification.

---

# Stage Information

**Stage ID**

Unique stage identifier.

Example:

```
Stage 00
```

---

**Stage Name**

Human-readable stage name.

Example:

```
Project Zero
```

---

# Purpose

Why does this stage exist?

Describe the engineering objective of this stage.

A stage must have exactly one primary purpose.

---

# Inputs

What information or artifacts are required before this stage can begin?

Inputs may include:

- Documents
- Decisions
- Requirements
- Previous stage outputs

A stage cannot start until all required inputs are available.

---

# Participants

Which engineering roles participate in this stage?

Examples:

- Engineering Leader
- Software Architect
- Backend Engineer
- Research Engineer
- QA Engineer

Only required participants should be included.

---

# Activities

Describe the engineering work performed during this stage.

Activities should be written as actions.

Example:

- Analyze the problem.
- Identify stakeholders.
- Evaluate technical constraints.
- Produce architectural alternatives.

Activities explain **how** the objective is achieved.

---

# Deliverables

What artifacts must exist before this stage is considered complete?

Examples:

- Project Charter
- Architecture Diagram
- Research Report
- ADR
- Technical Decision

Deliverables represent measurable outputs.

---

# Definition of Done

A stage is complete only when every completion criterion has been satisfied.

The Definition of Done must be objective.

Avoid subjective statements.

Examples:

✔ Architecture approved.

✔ Risks documented.

✔ Stakeholders identified.

✘ "Looks good."

---

# Next Stage

Which stage begins after successful completion?

Only one next stage should be defined.

---

# Engineering Notes

Optional observations.

Engineering Notes provide context without changing the execution process.

---

# Stage Review

Before closing the stage, verify:

- Was the objective achieved?
- Are all deliverables complete?
- Does the Definition of Done pass?
- Can the next stage begin without uncertainty?

Only then may the Engineering Engine progress.

---

# Engineering Principle

A stage exists to reduce uncertainty.

If uncertainty remains, the stage is not complete.
